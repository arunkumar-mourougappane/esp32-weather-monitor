# Release Notes — v3.1.0

This is a patch/stability release focused on battery indicator improvements and four root-cause fixes for boot crashes, network failures, and silent input regressions introduced in v3.0.0.

---

## 🚀 Highlights

* **Reworked battery indicator** — hardware-accurate LiPo discharge curve, charging bolt icon, LOW badge, dashed fill, and a runtime-estimate row in Settings
* **Boot crash eliminated** — dedicated 24 KB FreeRTOS task for app logic avoids `loopTask` stack overflow during TLS handshake
* **NTP now succeeds on first attempt** — SNTP mode fix + race-condition fix; NTP syncs in < 2 seconds instead of exhausting all retries
* **AQI data now populated** — corrected invalid API parameter (`weed_pollen` → `ragweed_pollen`); AQI and pollen counts were zero in every prior release
* **Touch / swipe fully restored** — `M5.update()` race between `loopTask` and `InputTask` was silently clearing all touch edge flags

---

## ✨ New Features

### LiPo-Accurate Battery Percentage
`DisplayManager::_lipoMvToPercent()` replaces the previous linear voltage-to-percent mapping with a piecewise-linear interpolation table tuned to a real LiPo cell's knee-shaped discharge profile. The result is an accurate state-of-charge reading across the full battery range rather than a linear approximation that over-reports at mid-range and under-reports near empty.

### Battery Charging Indicator
When USB power is detected (`packMv > 4050 mV`), the battery widget in the clock strip renders a bold bolt symbol (`+` shape, cross-bar fill) instead of a fill bar, and the percentage label is prefixed with `+` (e.g. `+82%`). The Settings diagnostics row appends `"Charging"` to the voltage string.

### LOW Badge in Clock Strip
When `_cachedBatLevel ≤ 15 %` and the device is not charging, an inverted `LOW` badge (42 × 14 px) appears in the clock strip to the left of the time. If the `RAIN` badge is also active, `LOW` shifts right by 58 px so both coexist without overlap.

### Dashed Battery Fill at ≤ 15 %
Below 15 %, the battery bar renders in an alternating-column dashed pattern (every other pixel column drawn, creating a visual striped effect) to reinforce the LOW badge warning in a second visual channel.

### ADC Deduplication
`DisplayManager::sampleBattery()` averages 4 consecutive `analogReadMilliVolts(35) × 2` readings before updating `_cachedBatVoltage`, `_cachedBatLevel`, and `_cachedBatCharging`. This reduces ADC noise on the ESP32 12-bit SAR ADC, which can vary by ±50–150 mV across individual samples.

### Battery Runtime Estimate
`AppController` maintains a rolling ring buffer (`RTC_DATA_ATTR int32_t rtcBatRing[8]`) of battery voltage samples across deep-sleep wakeup cycles. Once at least two samples are available, a per-sample drain rate is computed and extrapolated to a remaining-hours estimate, which is stored in `WeatherData::batteryRuntimeH`. The Settings diagnostics page displays an `"Est. Runtime: ~N h left"` row when the estimate is non-zero.

## 🐛 Bug Fixes

### `loopTask` Stack-Canary Crash on Every Boot
**Root cause:** The mbedTLS TLS handshake requires ~8–10 KB of stack. The Arduino `loopTask` has a fixed 8 KB stack enforced by `sdkconfig.h`, which overrides the `-D CONFIG_ARDUINO_LOOP_STACK_SIZE` build flag applied in `platformio.ini`. Running the TLS fetch path in `setup()` (which runs on `loopTask`) triggered a `Guru Meditation Error: Stack canary watchpoint triggered (loopTask)` crash on every boot.

**Fix:** `setup()` spawns a dedicated `_appTask` via `xTaskCreatePinnedToCore()` with 24 KB of stack pinned to core 1 at priority 5. `setup()` returns immediately; all normal-mode firmware logic runs inside `_appTask`. `loopTask` stays idle in normal mode (no `loop()` body runs).

### NTP Always Failing on Every Boot
**Root cause (1 — SNTP mode):** The default SNTP sync mode is `SNTP_SYNC_MODE_SMOOTH`, which uses gradual `adjtime()` time steps. In smooth mode, `SNTP_SYNC_STATUS_COMPLETED` is only set after the adjustment fully converges — which can take minutes. In practice the status cycled between `IN_PROGRESS` and `RESET` for the entire 5-second per-attempt timeout on all 3 retry attempts, so NTP never succeeded and the BM8563 RTC fallback was used on every single boot.

**Root cause (2 — race condition):** Even when a sync completed, the post-loop `if (esp_sntp_get_sync_status() == SNTP_SYNC_STATUS_COMPLETED)` check re-queried the volatile SNTP daemon status. If the SNTP task had already advanced to its next cycle and reset the status flag, the outer check saw `RESET` despite the sync having succeeded, and treated the attempt as a failure.

**Fix:** Added `esp_sntp_set_sync_mode(SNTP_SYNC_MODE_IMMED)` before `configTzTime()` so the sync completes in a single step. Refactored the polling loop to capture `synced = true` inside a local `bool` the instant `status == SNTP_SYNC_STATUS_COMPLETED`; all subsequent checks use that `bool` rather than re-querying the volatile SNTP state.

### AQI Always HTTP 400 / All Pollen Counts Always Zero
**Root cause:** The Open-Meteo Air Quality API URL contained `weed_pollen` as a query parameter. This is not a valid variable name for the open-meteo air-quality endpoint — the correct parameter is `ragweed_pollen`. Every AQI request returned HTTP 400 Bad Request, leaving `data.aqi = 0` and all pollen counts at zero in every release prior to v3.1.0.

**Fix:** Changed the URL query string and the JSON response parse key from `weed_pollen` to `ragweed_pollen`. Error response body is now logged on 4xx failures to aid diagnosis of any future parameter issues.

### `setSocketOption` Errno 9 / Connection State Corruption
**Root cause:** `http.setReuse(true)` was set on the shared `HTTPClient` instance. This left the underlying TLS socket file descriptor open after `http.end()`. When the next fetch targeted a different host (the five fetches span `googleapis.com`, `air-quality-api.open-meteo.com`, and `api.open-meteo.com`), the HTTP client attempted to reuse the stale fd on a connection to a different socket, producing `setSocketOption(): fail on 0, errno: 9 (EBADF)` log noise and corrupting connection state for subsequent fetches.

**Fix:** Removed `http.setReuse(true)`. Added `client.stop()` before each `http.begin()` call that switches to a different host (fetches 3, 4, and 5), ensuring a clean TCP/TLS socket on every host transition.

### Touch Events Silently Dropped After FreeRTOS Task Change
**Root cause:** Introducing `_appTask` caused `setup()` to return, making `loop()` run normally on `loopTask` (core 1). `loop()` called `M5.update()` at the full FreeRTOS idle rate. `InputTask` (core 0) independently called `M5.update()` every 20 ms inside `_processTouchGestures()`.

`M5.update()` is not thread-safe: the `wasPressed()` / `wasReleased()` edge flags inside the M5Unified touch-state machine are one-shot — they are set on the rising edge and cleared on the first read. `loopTask` was consistently winning the race and clearing every flag milliseconds before `InputTask` polled, so all swipes and taps were silently dropped. This regression affected every touch interaction — forecast swipes, settings taps, and the swipe-up overlay.

**Fix:** Removed `M5.update()` from `loop()`. `InputTask` exclusively owns the 50 Hz `M5.update()` polling cadence — there is never a correct reason for a second caller. Added `delay(10)` to `loop()` so `loopTask` yields the CPU rather than busy-spinning.

## 🔄 Changed Behaviour

| Area | Old behaviour | New behaviour |
|---|---|---|
| Battery % | Linear voltage → % | Piecewise-linear LiPo discharge curve |
| Battery fill ≤ 15 % | Solid thin fill bar | Alternating-column dashed pattern |
| Battery level ≤ 15 % | % text only | `LOW` badge + `%!` label suffix |
| Battery charging | Fill bar (no indication) | Bolt symbol + `+N%` label |
| Settings battery row | `x.xx V  (N%)` | `x.xx V  (N%)` or `x.xx V  (N%) Charging` |
| NTP sync | ~15 s / 3 retries / always falls back to RTC | < 2 s / succeeds on first attempt |
| AQI | Always 0 (HTTP 400) | Valid EPA AQI index returned |
| Pollen (grass/birch/weed) | Always 0 (HTTP 400 upstream) | Valid hourly pollen counts from open-meteo |
| Touch / swipe | Silently dropped in normal mode | Fully functional |
| App execution path | Runs in `loopTask` (8 KB stack) | Runs in `_appTask` (24 KB stack, core 1) |

## ⬆️ Upgrade Notes

1. No re-provisioning required — NVS layout is unchanged.

3. If you observed persistent `[02] NTP sync failed` diagnostics under v3.0.0, this is resolved.
4. The `APP_VERSION` build flag in `platformio.ini` is now `3.1.0`.
